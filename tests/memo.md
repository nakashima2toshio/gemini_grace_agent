#### 

1. Files （ファイル単位のサマリー）
   これがデフォルトの表示で、プロジェクト全体の進捗を確認するのに適しています。

* 何が表示されるか: 各 .py ファイルごとのカバレッジ率。
* 主な列の意味:
  * Statements: 実行可能なコード行の総数（コメントや空行を除いたもの）。
  * Missing: テスト中に一度も実行されなかった行の数。ここが 0 ならそのファイルは完遂しています。
  * Excluded: # pragma: no cover などで意図的に計測から除外した行数。
  * Coverage: そのファイルの網羅率（％）。
* 活用方法:
  「どのファイルが一番テスト不足か？」を特定します。ファイル名をクリックするとソースコードが表示され、赤色で塗られた行が「テストされ
  なかった行」です。

---

2. Functions （関数・メソッド単位）
   どのロジック（関数）が動いていないかをピンポイントで探すのに便利です。

* 何が表示されるか: ファイル内の関数（def）やクラスメソッド単位のリスト。
* 主な特徴:
  * 一つのファイルの中に複数の関数がある場合、それぞれのカバレッジ率が個別に表示されます。
* 活用方法:
  * ファイル全体の率は高くても、「実は特定のエラーハンドリング用の関数が全く動いていない」といった状況を見つけ出せます。
  * 巨大なファイルの中で、優先的にテストすべき重要なロジックが漏れていないかを確認するのに最適です。

---

3. Classes （クラス単位）
   オブジェクト指向の設計において、機能（クラス）ごとの網羅率を確認できます。

* 何が表示されるか: class 定義ごとのリスト。
* 主な特徴:
  * クラス内に定義されたすべてのメソッドを合算した統計が表示されます。
* 活用方法:
  * 「このサービス（クラス）は複雑なのにテストが少ない」といった、設計コンポーネント単位での品質チェックに使います。
  * 特に services/ フォルダ内のビジネスロジックをクラスで管理している場合、ここを見ると機能単位の完成度が分かります。

---

レポート内の「色」の意味
詳細ページ（コード表示）では、行の左側が以下の色で塗り分けられています：

* 緑色 (run): テストで実行された行。
* 赤色 (missing): テストで一度も実行されなかった行（＝ここを通るテストケースが不足している）。
* 黄色/オレンジ (partial): if 文などで、一方は通ったがもう一方（else 側など）が通っていない「条件分岐の片漏れ」。
* 灰色 (excluded): 計測対象外。

まとめ

* まず Files で全体を見て、率の低いファイルを探す。
* 次に Functions やコード詳細で 赤色の行 を確認する。
* その赤色の行を実行するためのテストケースを tests/ に追加する。
